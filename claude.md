# ТЗ: Исправление багов Telegram-бота (GoodHouse Bot)

## Контекст проекта

Telegram-бот для строительной компании.
Отслеживает обращения клиентов в групповых чатах,
контролирует SLA (время ответа), фиксирует обещания сотрудников,
эскалирует нарушения по цепочке ответственных.

Стек: Python, aiogram 3.x, PostgreSQL, psycopg2.
Хостинг: Яндекс Клауд. Репозиторий: GitHub.

## Структура файлов

- `bot.py` — основной файл бота (весь код)
- `db.py` — функции подключения к БД
- `config.py` — токен, DSN, константы (не в репозитории)

## БЛОК 1. SLA (реакция на сообщение клиента)

### 1.1 Закрытие SLA при ответе сотрудника

Любой сотрудник из USERS написал сообщение в чат после клиента
→ SLA немедленно закрывается (сбрасываются все ping*_at в sla_state_v1).

Поле в БД: `answered_at`, `answered_by_user_id` в таблице `sla_state_v1`.

Watcher перед каждым пингом проверяет:
если `answered_at >= last_incoming_at` → пропустить чат.

### 1.2 Интервалы SLA

**prebuild:**
- ping1: через 40 мин → менеджер
- ping2: +15 мин → директор

**build / warranty:**
- ping1: через 90 мин → прораб или нач.участка (см. 1.5)
- ping2: +30 мин → директор по строительству
- ping3: +30 мин → директор

### 1.3 Игнор-фразы клиента

Если клиент пишет одну из фраз:
спасибо, ок, окей, хорошо, понял, поняла, понятно,
мы подумаем, подумаем, посмотрим, хорошо посмотрим,
ладно, договорились, принято, благодарю

→ Сообщение записывается в БД как обычно.
→ SLA НЕ запускается и НЕ сбрасывается.

Реализация: функция `is_client_ignored_phrase(text) -> bool`.

### 1.4 Учитывается только последнее сообщение клиента

SLA считается только по последнему `direction='in'` в чате.
При новом входящем — все ping*_at сбрасываются.

### 1.5 Динамическая цепочка эскалации для build

Прораб считается "в чате" если выполняется хотя бы одно условие:
- является членом Telegram-группы (get_chat_member → member/administrator/creator)
- ИЛИ когда-либо писал в этот чат (есть запись в таблице messages)

Если прораб в чате есть:
→ прораб → директор по строительству → директор

Если прораба нет:
→ нач.участка → директор по строительству → директор

Реализация: `escalation_chain_dynamic(bot, conn, chat_id, stage)` — async функция.

### 1.6 Проверка участника перед отправкой

Перед каждым пингом: `get_chat_member(chat_id, user_id)`.
Если пользователя нет → пропустить, перейти к следующему уровню.

### 1.7 Закрытие SLA по реакции сотрудника

Сотрудник поставил реакцию на последнее сообщение клиента
→ SLA закрывается так же как при ответе.

Handler: `@router.message_reaction()` (aiogram 3.x).

## БЛОК 2. Promise (обещания сотрудников)

### 2.1 Детект обещания

Обещание фиксируется только если в сообщении сотрудника есть
одно из ключевых слов:
уточню, скажу, вернусь, отпишусь, пришлю, дам ответ,
сообщу, напишу, дам знать, сориентирую, перезвоню, скину,
позже отвечу, отвечу позже, позже напишу

### 2.2 Срок обещания — только при явном временном контексте

Срок парсится ТОЛЬКО если в тексте есть:
сегодня, завтра, утром, вечером, днём,
до 15:00 / до 15 / к 18,
через 20 минут / через 2 часа / в течение часа,
день недели (вторник, ср, пт и т.д.)

Голые цифры НЕ являются сроком:
50к, 20 000, 2 окна, 3 варианта → срок не найден → needs_due_at = True.

Реализация: функция `_has_time_context(text) -> bool`.

### 2.3 Если срок не найден

Бот отвечает сотруднику:
"Ты написал «...», но не указал срок.
Напиши срок одним сообщением. Пример: «до 13:00» или «завтра до 12:00»."

Ждёт ответа 10 минут (PENDING_DUE_TTL).

### 2.4 Автозакрытие promise после выполнения

Обещание закрывается автоматически если сотрудник
(тот же user_id, тот же chat_id) отправил содержательное сообщение
после сообщения-обещания (message_id > promise.message_id).

Содержательное сообщение = любое из:
текст, документ, фото, видео, голосовое, аудио,
пересланное сообщение, caption, стикер.

Реализация: `_close_last_promise(db, chat_id, user_id, current_message_id)`.

SQL при закрытии:
```sql
UPDATE promises
SET is_done = TRUE,
    done_at = NOW() AT TIME ZONE 'utc'
WHERE id = %s
```

### 2.5 Новое обещание вместо выполнения старого

Если сотрудник написал новое обещание, а старое ещё открыто:
→ старое закрывается (is_done = TRUE)
→ новое создаётся как обычно

### 2.6 Таймлайн напоминаний по просрочке

- за 10 мин до срока → личка автору
- +5 мин просрочки → личка автору
- +20 мин → директору (OWNER_USER_ID)
- +40 мин → директору (финальная эскалация)

## БЛОК 3. Пауза чата

### 3.1 Команды

`/pause` — поставить чат на паузу
`/resume` — снять паузу

Только для сотрудников. Только в групповых чатах.

### 3.2 Поведение при паузе

- SLA-эскалации: отключены
- Контроль обещаний: работает как обычно
- Контроль тишины: порог меняется с 72 часов на 7 дней

### 3.3 Напоминание через 7 дней

Через 7 дней после постановки на паузу — личка сотруднику
который поставил паузу:
"Ты поставил чат на паузу 7 дней назад.
Проверь — может, пора написать клиенту? /resume"

Отправляется только один раз (поле pause_reminded_at).

### 3.4 Таблица в БД
```sql
CREATE TABLE IF NOT EXISTS chat_pause_state (
    chat_id           BIGINT PRIMARY KEY,
    is_paused         BOOLEAN NOT NULL DEFAULT FALSE,
    paused_at         TIMESTAMPTZ,
    paused_by         BIGINT,
    resumed_at        TIMESTAMPTZ,
    pause_reminded_at TIMESTAMPTZ,
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## БЛОК 4. Контроль тишины

### 4.1 Логика

Нет сообщений от компании (direction='out') дольше порога → пинг.

Без паузы: порог 72 часа.
С паузой: порог 7 дней.

1 напоминание на период — не спамить.
Факт отправки фиксировать в silence_state_v1 (ping1_at и т.д.).

### 4.2 Цепочки по стадиям

**prebuild:** менеджер → директор
**build:** прораб → нач.участка → дир.строительства → директор
**warranty:** нач.участка → дир.строительства → директор

### 4.3 Частота watcher

Каждые 10 минут (`asyncio.sleep(600)`).

## БЛОК 6. Что НЕ трогать

- Логика записи всех сообщений в таблицу `messages` — не менять
- Таблицы `chats`, `users`, `sla_ping_events` — структура не меняется
- `db.py` и `config.py` — не трогать
- Токен и DSN — всегда в `config.py`, никогда не в коде

**Обращение с секретами (жёстко — правило нарушалось многократно):**

* `dostup.md`, `доступы.md` и любые gitignored-файлы с секретами — справочник ДЛЯ ЧЕЛОВЕКА, НЕ операционный источник. Их НЕ читать/грепать/катать для операций ВООБЩЕ — ни `grep`/`cat`/`echo` в терминале, ни Read-инструментом (Read кладёт содержимое в транскрипт сессии = утечка; маскировка формат/кириллицу не ловит).
* Секреты для операций брать ТОЛЬКО из рабочего источника: `.env` на самом сервере, переменные окружения, ssh-config. НЕ из файла-справочника.
* Нужен НОВЫЙ секрет (ротация/генерация) — генерировать НА сервере (`openssl rand …`), писать прямо в `.env`, значение НИГДЕ не печатать/не выводить в терминал/лог.
* Нужного значения нет на сервере — СПРОСИТЬ владельца, а не доставать из файла-справочника.
* Записать новое значение В файл-справочник (`dostup.md`) — можно точечно Write/Edit, но БЕЗ чтения/грепа файла и БЕЗ вывода значения.
* Секрет, попавший в терминал/лог/транскрипт = СКОМПРОМЕТИРОВАН → ротировать у провайдера, вычистить исторические логи, проверить внешние агрегаторы (Sentry), уведомить владельца. Не замалчивать.
* Внешние логгеры (`httpx`/`httpcore` и подобные) — на уровне WARNING: на INFO они пишут полные URL с токенами (`…/bot<token>/…`).
* Цель — перенести секреты в менеджер паролей (Bitwarden, бесплатно), чтобы grep-able plaintext-файла не осталось.

## БЛОК 7. Порядок проверки после деплоя

1. Клиент пишет → сотрудник отвечает → эскалации нет
2. Клиент пишет "спасибо" → SLA не запускается
3. Сотрудник пишет обещание с файлом → promise закрывается
4. Сотрудник пишет "50к" → promise не создаётся
5. /pause → клиент молчит → SLA нет → через 7 дней напоминание
6. Убрать прораба из группы → эскалация идёт нач.участку
